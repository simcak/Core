Auth serviceWhen a user logs in, they send their password here. It checks credentials, then returns a token (JWT) — a signed string the client includes in every future request to prove identity. No other service needs to know the password, only the token.
API serviceThe main brain. A client (browser, phone) sends a request — "give me all the books". The API checks the token, queries the database, and sends back JSON. Your /catalog/ service is one of these.
DatabasePersistent storage. Everything that needs to survive a restart lives here. PostgreSQL, MySQL, SQLite are popular choices. Your app uses Alembic to manage its schema (the shape of its tables).
Queue / workerSome jobs are too slow to do inline — sending emails, resizing images, generating reports. Instead you drop a task into a queue (like Redis or RabbitMQ) and a worker picks it up in the background. The user never waits.
File storageDatabases are bad at storing big blobs like images and videos. Instead you upload files to object storage (AWS S3, Cloudflare R2) and only store the URL in the database.
CacheRedis or Memcached. If computing something takes 200ms (a complex DB query), you store the result for 60 seconds. The next 1000 requests get it instantly. Caches are temporary — they can be wiped and rebuilt.
This is the most important architectural decision you'll make early on. Neither is wrong — they have different tradeoffs.
Monolith start here
One codebase, one deployable, one database.

Auth, catalog, audio — all in the same app.

✓ Easy to build and reason about
✓ Easy to debug (one log stream)
✓ No network calls between services
✓ One database transaction across features
✗ Hard to scale one part independently
✗ A crash takes everything down
Microservices later, if needed
Many codebases, each deployed separately.

Your project is already doing this.

✓ Scale hot parts independently
✓ Teams own separate codebases
✓ One service crashes, others keep running
✗ Much more complex to operate
✗ Network failures between services
✗ Hard to do cross-service transactions
Rule of thumb: Start with a monolith. Split into services only when you have a specific, proven reason — a part is too slow to scale with the rest, or separate teams need to work independently. Most apps never need microservices.

Client layerYour React app (or native iOS/Android app). It only knows how to make HTTP requests and render UI. It has no business logic — it sends requests and shows what comes back. This layer is stateless on the server; its state lives in the browser.
API gateway / reverse proxyThe single entry point for all traffic. It routes /api/auth to the auth service, /api/catalog to the catalog service, and / to the frontend. It also handles SSL termination, rate limiting, and load balancing. Your /nginx/ folder does this.
Business logic layerThis is the "real" backend. It enforces rules — "a user can only delete their own books", "a free account can upload max 3 files". It reads/writes the database and calls other services. Keep all rules here, never in the client.
Catalog serviceManages books data. It exposes REST endpoints like GET /books, POST /books. Other services call it over HTTP just like a browser would.
Audio serviceHandles audio files, transcription, or playback metadata. Isolated so audio bugs don't affect auth or catalog.
Data layerThe database. Only the business logic layer talks to it directly — never the client. Alembic migrations define the schema and track changes over time, so your database evolves safely as your app grows.
Cache layerRedis sits in front of the database for frequently-read data. Hot data (a user's profile, a popular book listing) is served from memory in microseconds instead of hitting the database every time.
The key insight: if all your business logic lives behind an API, every client (web, iOS, Android, Watch) is just a different way to call the same endpoints. The backend doesn't know or care what's calling it.

1
Never put logic in the frontend. If the frontend calculates something — a price, a permission check — it'll have to be duplicated in every future client. Keep all rules in the API.
2
Use token-based auth (JWT). Cookies are browser-only. JWTs work on iOS, Android, watches, CLI — any client just puts the token in an Authorization header.
3
Version your API. Prefix all routes with /api/v1/. When you change a response shape, you can add /v2/ without breaking old clients still on /v1/.
4
Return data, not views. The API returns JSON. Each client decides how to render it. Never return HTML from the API.
5
Document your API. Use OpenAPI / Swagger. When a native dev joins later, they can work independently from the spec without asking you how every endpoint works.
Follow one request from the user's finger to the database and back.
Scenario: a logged-in user taps "search for books" on the web app.
1
Client sends a request. The browser makes GET /api/v1/catalog/books?q=harry with an Authorization: Bearer <token> header.
2
nginx receives it. It sees /api/v1/catalog and forwards the request to the catalog service. The browser doesn't know this address exists.
3
Catalog service validates the token. It calls the auth service: "is this token valid?". Auth confirms: "yes, this is user #42". Catalog now knows who is asking.
4
Check the cache. Has someone searched "harry" in the last 30 seconds? If yes, return the cached result instantly — no database needed.
5
Query the database. Cache miss — catalog runs SELECT * FROM books WHERE title ILIKE '%harry%' on PostgreSQL. Gets 7 results.
6
Store in cache, return response. Saves the result in Redis for 30s. Returns JSON to nginx, which passes it back to the browser.
7
React re-renders. The component receives the JSON, maps over the array, and displays the 7 books. Total time: ~50ms.
