System Architecture

AudioBerry is a microservices-based audiobook platform. All external traffic enters through Nginx, which routes requests to the appropriate backend service. Services share a single PostgreSQL instance using separate schemas. Object storage (MinIO) holds audio files, book cover images, and user avatars.

Dev shortcut: Run make up for the full stack on http://localhost:8000 (HTTP → HTTPS redirect) or https://localhost:4443. pgAdmin is at http://localhost:5050. MinIO console at http://localhost:9001.

Startup Order (docker compose)

Services start in dependency order enforced by health-checks:

  1. PostgreSQL — waits until pg_isready passes
  2. MinIO → minio-init (creates buckets: audio-files, covers, avatars)
  3. auth-migrate / catalog-migrate / audio-migrate (Alembic upgrade head)
  4. auth-seed / catalog-seed / audio-seed (dev only)
  5. Auth, Catalog, Audio services (wait for their migrate job)
  6. Frontend, Nginx (wait for services)

Microservices

Each backend service is an independent FastAPI application running in its own Docker container. Services communicate externally through Nginx — they do not call each other directly.

API Reference

All endpoints are accessed through Nginx. The base URL in production is https://localhost:4443 (or the configured domain). Click any endpoint to expand details.

Authentication: The auth service uses HttpOnly cookies (access_token, refresh_token). Cookies are set on login and sent automatically with each request. No manual header needed from the frontend — Nginx forwards them. Swagger UI is at /auth/docs.

Public API

prefix: /api/public/ Swagger: /api/public/docs

Key-gated surface, separate from the session cookies. Send X-API-Key: <key> or Authorization: Bearer <key>. Nginx rate limits every /api/public/* route to 10 r/m per IP (429 over limit). An aggregated Swagger UI at /api/public/docs merges the auth, catalog and audio OpenAPI schemas.

Authentication Flows

The auth service issues short-lived JWT access tokens and longer-lived refresh tokens. Both are stored as HttpOnly cookies set by the auth service and forwarded automatically by the browser on every request through Nginx.

📧 Email + Password Login
1
User submits credentials
POST /auth/login { email, password }
2
Auth service validates
Looks up user by email, verifies bcrypt hash
3
Issues JWT access token
Signed JWT with user_id + role. Short-lived (minutes).
4
Issues refresh token
Random opaque token stored hashed in DB. Long-lived (days).
5
Sets HttpOnly cookies
Set-Cookie: access_token=...; HttpOnly; Secure; SameSite=Lax
Set-Cookie: refresh_token=...; HttpOnly; Secure; SameSite=Lax
6
Frontend stores nothing
Cookies sent automatically by the browser on all subsequent requests.
🔐 42 OAuth Login
1
User clicks "Login with 42"
GET /auth/42
2
Redirect to 42 authorize URL
Authlib builds the authorization URL and redirects the browser.
3
42 redirects back to callback
GET /auth/42/callback?code=...
4
Exchange code → fetch 42 profile
Gets id, email, login from api.intra.42.fr/v2/me
5
Upsert user in DB
Find by oauth_id → link if email matches → create new. Marks email_verified=true.
6
Set cookies + redirect to frontend
Same cookie mechanism as email login. Redirects to configured frontend URL.
🔄 Token Refresh
1
Access token expires
API returns 401. Frontend detects this.
2
Frontend calls refresh endpoint
POST /auth/jwt/refresh
3
Service validates refresh token cookie
Checks token_hash in DB, not expired, not revoked.
4
Issues new access token
Sets new access_token cookie. Refresh token is NOT rotated on every refresh.
🎯 Request Authorization
1
Browser sends request with cookies
access_token cookie attached automatically.
2
Nginx forwards to correct service
No auth check at Nginx level — services validate independently.
3
Service runs verify_jwt_access() dependency
Reads access_token cookie, verifies signature, extracts user_id + role.
4
Role check (if required)
verify_jwt_access(["admin"]) — 403 if role not in list.
verify_jwt_access(["publisher","admin"]) — for publisher endpoints.

Nginx Routing

Nginx is the single entry point for all traffic. It terminates TLS, enforces client_max_body_size 512m (for audio uploads), and routes by URL prefix to the correct upstream. HTTP on port 80 redirects to HTTPS on port 443.

Port mapping: Nginx listens on container port 80/443, exposed to host as 8000:80 and 4443:443. The dev stack uses self-signed certs — accept the browser warning or add the cert to trust store.

Dev Mode — Frontend Outside Docker

Running make frontend starts Vite on localhost:5173 outside Docker. The Nginx config switches to nginx.dev.conf, which has no frontend upstream. Vite proxies /api to localhost:8000. CORS origins must include http://localhost:5173 in .env.

Metrics Scraping

Each backend service exposes /metrics via prometheus_fastapi_instrumentator. Prometheus (when running) scrapes all three services directly on the internal Docker network — it does not go through Nginx.
